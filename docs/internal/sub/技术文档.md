## 1. 技术背景

### 1.1 项目简介

> **项目缘起**：AgentHub 是一项为期 20 天的 AI 全栈挑战赛实战项目。借助 Trae 等 AI Coding 工具进行人机协作开发，覆盖需求、架构、三端实现到联调交付的完整流程（协作过程见配套《AI 协作开发记录》）。

AgentHub 是一个以 **IM 聊天为核心范式**的多 Agent 协作平台。技术形态上，它是 **AI Runtime Platform**——不是普通的 CRUD + Chat UI，真正复杂度集中在 **event lifecycle、runtime state、execution isolation、stream protocol** 四件事上。

系统采用 **React（前端）+ Go（后端）+ Python（Agent 端）** 三层架构——后端只管状态与中继、Agent 端唯一负责执行；跨端协议由 **契约层（contracts）** 的 YAML 单一定义、通过 `make generate` 指令生成三端类型；三类编码 Agent（Claude Code / OpenCode / Codex）经统一 Adapter 接入，由 Orchestrator 在群聊中拆解分派——新增 Agent 零改前端。

**工程化**：三端代码以 **pnpm monorepo** 同仓组织；**husky** 在 pre-commit 经 lint-staged 分语言强制校验——前端 ESLint + Prettier、后端 gofmt + goimports、Agent 端 ruff；检测到契约文件变更时自动 `make generate` 重生成三端类型并要求 secret 二次确认，从源头杜绝契约漂移；commit-msg 经 commitlint 强制 Conventional Commits + scope 限定（frontend / backend / agentend / common / docs / other）。

### 1.2 技术挑战

Runtime 系统真正最难的从来不是 prompt engineering，而是以下工程问题：

| 挑战 | 本质 | AgentHub 对应 |
|------|------|---------------|
| **HTTP 连接 = Agent 生命周期** | 长任务下连接断开即任务丢失、无法重连 / 回放 | 引入 Redis Stream 做持久缓冲 + Hub 实时推送，解耦连接与生命周期 |
| **状态一致性** | Go 与 Python 状态漂移导致「幻觉态」 | 冻结状态迁移权威：仅 Python 产生执行态迁移 |
| **事件顺序** | 流式事件乱序 / 重复 / 丢失 | 序列号（sequence）严格递增 + Last-Event-ID 回放 + 前端去重 |
| **流持久性** | 刷新 / 重连后历史不丢 | 三级存储：Hub（实时）→ Redis Stream（短期回放）→ MySQL（长期） |
| **多 Agent 隔离** | 并行 Agent 互相污染工作区 | 每 Agent 独立 git worktree + 独立分支 |
| **协议漂移** | 三端类型定义手工同步易腐化 | 契约层 YAML 单一来源 + `make generate` 三端代码生成 |

### 1.3 技术目标

| 优先级 | 目标 | 衡量标准 |
|--------|------|----------|
| P0 | 三层职责冻结、不可越界 | Go 不执行 Agent、Python 不持有持久化权威（见 §2.3） |
| P0 | 流式全链路不丢不重 | 断线重连后事件序号严格递增，无重复 |
| P0 | Agent 可插拔 | 新增 Agent 仅在 Adapter 层 + Registry 注册，前端零改动 |
| P1 | 流式渲染 60fps | rAF 批处理 + 虚拟列表，长会话不卡 |
| P1 | 跨端类型零漂移 | 契约变更必须经 `make generate` 同步三端，并在 `contracts/logs/` 记录变更日志 |
| P2 | 工作区物理隔离 | git worktree 互不污染，可独立 commit / revert |

## 2. 技术架构概览

### 2.1 总体架构

AgentHub 从传统的 **Request-Driven** 演进为 **Event-Driven Runtime**：HTTP 请求只负责触发，Agent 产出以事件流（EventEnvelope）形式贯穿三端，前端以 reducer 模式从事件流派生 UI。

```
┌──────────────────────────────────────────────────────────────────────┐
│                  React Frontend (SPA, :5173)                          │
│   EventSource ──▶ useChatStream ──▶ messageStore(rAF) ──▶ Cards       │
│        ▲                                  (reducer over events)       │
└────────┼───────────────────────────────────────────────────────────────┘
         │ ① SSE GET /api/tasks/:id/stream?session_id&message_id
         │   ③ REST POST /api/tasks/:id/run  /review  /messages ...
┌────────┴───────────────────────────────────────────────────────────────┐
│              Go Backend — Platform State Layer (:8080)                  │
│   controller → service → dao(GORM)        │  stream/{hub, writer}      │
│   • 只 record / observe / request          │  • Hub 内存 Pub/Sub(实时)  │
│   • 不执行 Agent、不持执行态权威            │  • Redis Stream(回放/重连) │
└────────┬───────────────────────────────┬──┴────────────────────────────┘
         │ ② SSE POST /v1/agent/stream   │ ④ MySQL(长期) + Redis(短期)
┌────────┴───────────────────────────────┴────────────────────────────────┐
│          Python AgentEnd — Execution Layer (:8001)                      │
│   FastAPI → AdapterRegistry → {Claude/OpenCode/Codex/Orchestrator}      │
│   • 唯一执行态权威（task.running/completed/failed）                      │
│   • LangGraph 编排引擎 · ExecutionEngine 波次并行 · git worktree 隔离     │
└──────────────────────────────────────────────────────────────────────────┘
                  ▲ ⑤ contracts/schemas/*.yaml  ──make generate──▶  三端 generated/
```

**核心数据流（单聊）**：前端发 REST 触发任务 → 后端创建消息、建立 SSE 通道、调用 Agent 端流式接口 → Agent 端按事件流式回传 → 后端双写（Hub 实时推前端 + Redis/MySQL 持久化）→ 前端按事件增量渲染。

### 2.3 架构设计原则（冻结边界）

> 以下 6 条是**不可违反的硬边界**，比任何功能都重要，源自三层架构设计。核心纪律：**Runtime for Agents，不是 Generic Distributed Runtime**——永远优先服务 Agent 执行场景，不坠入「做基础设施 > 做 AI 产品」陷阱。

| 冻结 | 原则 | 禁止 |
|------|------|------|
| **冻结 1** | Go 只 record / observe / request，**永不执行 Agent**；Python 只 produce 执行迁移，**永不拥有持久化权威** | 后端变成第二个执行引擎 |
| **冻结 2** | **仅 Python** 能产生执行状态迁移（running/completed/failed/cancelled）。Go 与 Python 状态不一致时，Go 通过协议请求 Python 自行终结，而非强制改 | Go force transition |
| **冻结 3** | EventBus 只是**进程内临时分发器** | ack / retry / durable queue / exactly-once / consumer group |
| **冻结 4** | Artifact 只是**任务产物的引用** | artifact DAG / dependency / versioning / lineage |
| **冻结 5** | LangGraph 只是**规划引擎**（plan/dispatch/aggregate/replan） | 接管 lifecycle / subscription / event ordering / runtime state |
| **冻结 6** | EventEnvelope v1 ABI 已冻结（type/category/sequence/terminal/render.kind） | 字段重命名；演进只能 version 递增 + 新增 type + deprecated 旧 type |

## 3. 系统模块设计

> 三端各司其职、契约层横切贯穿，目录结构遵循各自子项目的 AGENTS.md，便于利用 AI Coding Agent（如 Claude Code）进行并行开发。下方「模块划分」对应到实际目录。

### 3.1 Frontend

消费 SSE 事件流并以 reducer 模式派生消息 / 卡片 UI；管理会话、Agent、技能、管理后台等视图状态；通过 REST 触发任务、审查、应用产物。

|------|------|------|

| 卡片家族 | `frontend/src/components/cards/` | DiffCard / HtmlCard / PlanCard / RuntimeStatus 等 12 类（Attachment / Coord / FinalSummary / Image / PlanReview / Preview / TaskFailure / Tool / Html / Plan / Diff / RuntimeStatus） |

### 3.2 Backend

#### 职责说明

平台状态层：Task / Session / Message 的 CRUD 与持久化；SSE 通道管理（实时 Hub + Redis 回放）；调用 Agent 端流式接口并双写；消息路由（@）、技能管理、工作区元数据、管理后台。

#### 模块划分

| 模块 | 目录 | 职责 |
|------|------|------|
| 入口 | `backend/cmd/server/main.go` | DI 组装、GORM AutoMigrate、优雅关停 |
| Controller | `backend/internal/controller/impl/` | 13 个控制器（HTTP 入参校验 + 响应封装） |
| Service | `backend/internal/service/impl/` | 14 个实现文件（11 个核心服务 + 3 个辅助模块：group_chat_window / stream_helper / task_route） |
| DAO | `backend/internal/dao/{gorm,mock}/` | 8 个数据访问接口 + GORM 实现 + 测试桩 |
| Model | `backend/internal/model/` | 9 文件 11 张表的 GORM 结构体 |
| Stream | `backend/internal/stream/` | hub.go（Pub/Sub）+ writer.go（双写） |
| 客户端 | `backend/pkg/agentend_client/` | 调用 Python 端的 HTTP 客户端（含流式） |
| 基础设施 | `backend/pkg/{db,redis,storage,qiniu}/` | MySQL / Redis 单例、存储抽象、七牛上传客户端 |

#### 分层约定

严格四层：`controller → service → dao → model`。Service 与 DAO 均以 interface 定义（`service.go` / `dao.go`），impl 为实现、mock 为测试桩，保证可测与可替换。响应统一封装 `vo.Response{code, data, msg}`。

### 3.4 Contracts

#### 职责说明

三端共享协议的**单一来源（Single Source of Truth）**：以 `schemas/*.yaml` 统一定义 SSE 事件、Agent 请求/响应/路由、会话状态机、消息持久化等跨端协议，经 `make generate` 一键生成 Frontend(TS) / Backend(Go) / AgentEnd(Python) 三端类型，从源头杜绝手工同步导致的协议漂移。契约层是横切三端的「第四模块」——不承载运行时逻辑，只定义类型边界。

#### 模块划分

| 模块 | 目录 | 职责 |
|------|------|------|
| 协议契约 | `contracts/schemas/` | 7 个 YAML 定义跨端协议（事件 / 请求响应 / 路由 / 会话状态机 / 消息 / Repo 校验），三端协议单一来源 |
| 变更审计 | `contracts/logs/` | 契约变更记录（`YYYY-MM-DD-<desc>.md`：原因 / 文件 / 差异 / 跨端影响） |
| 生成器 | `scripts/generate_contracts.py` | YAML → Python / TS / Go 三端类型 |

#### 目录结构

```
contracts/
├── schemas/                 # 7 个 YAML 契约（单一来源，仅此处手改）
│   ├── event-types.yaml     #   SSE 事件类型 + StreamEvent
│   ├── agent-request.yaml   #   AgentType 枚举 + 请求
│   ├── agent-response.yaml  #   响应（content + artifacts + usage）
│   ├── agent-routing.yaml   #   路由 + 群聊消息查询
│   ├── session-state.yaml   #   会话状态机 + 合法转换
│   ├── message.yaml         #   持久化消息
│   └── validate-repo-path.yaml
├── logs/                    # 变更审计（每次 schema 变更必记）
└── .watch-paths             # husky 监听路径（检测变更触发 make generate）
```

#### 生成与变更纪律

- **契约优先**：跨端协议类型只认 `schemas/*.yaml`；各端 `generated/` 由 `make generate` 生成，**手改必被覆盖**。
- **三端落点**：同一 schema 生成三端各一份 —— Python `agentend/src/generated/`、TS `frontend/src/generated/`、Go `backend/internal/generated/`。
- **变更留痕**：改 `schemas/` 必在 `logs/` 写记录（原因 / 文件 / 差异 / 跨端影响），对齐 §1.3「跨端类型零漂移」目标。
- **工程守卫**：husky pre-commit 经 `.watch-paths` 检测契约变更，自动 `make generate` 重生成三端类型并要求 secret 二次确认（见 §1.1 工程化段）。

## 4. Adapter 体系设计

> Adapter 层屏蔽 Claude Code / OpenCode / Codex / Orchestrator 的 CLI 与协议差异，对上层暴露统一的 `stream_chat` 接口，是「Agent 可插拔」原则的落点。

### 4.1 Adapter 抽象层

`agentend/src/adapters/base.py` 定义抽象基类 `BaseAgentAdapter`：

| 方法 | 职责 |
|------|------|
| `create_session(session_id)` | 创建 / 复用会话上下文 |
| `chat(session_id, message, **kw) -> AgentResponse` | 一次性执行（非流式） |
| `stream_chat(session_id, message, **kw) -> AsyncIterator[StreamEvent]` | 流式执行，按事件产出（**核心接口**） |
| `interrupt(session_id) -> bool` | 中断当前会话执行 |
| `destroy_session(session_id)` | 销毁会话、清理进程与上下文 |

所有 Adapter 把异构输出（JSON Lines / NDJSON / Thread-Turn-Item）统一归一化为契约定义的 `StreamEvent`（见 §8）。三类编码 Agent（Claude Code / OpenCode / Codex）均以**最高权限**运行——跳过所有 CLI 安全沙箱与权限确认（如 `--dangerously-skip-permissions`、`--dangerously-bypass-approvals-and-sandbox`），以减少人工干预、更好地模拟多 Agent 自主协作场景。安全边界由平台层的规划审查（§5）与 Diff 应用二次确认（§11.3）保障，而非依赖 CLI 内部沙箱。

### 4.2 Claude Adapter

- 文件：`agentend/src/adapters/claude.py`，类 `ClaudeCodeAdapter`。
- 调用：通过配置中的 `claude` CLI 路径以 `asyncio` 子进程执行，支持会话续接（resume）、最大轮数与允许工具过滤。
- 输出解析：逐行读取 JSON Lines，经 `_TYPE_MAP` 将 CLI 原始类型归一化为契约 `StreamEvent`：`system→init`、`tool_use→tool_call`、`tool_result→tool_result`、`result→done`；另通过 `stream_event/content_block_delta` 独立产出 `text` 增量事件。归一化后的 `StreamEvent` 由后端透传至前端，前端直接按事件类型渲染，无需二次映射。

### 4.3 Codex Adapter

- 文件：`agentend/src/adapters/codex.py`，类 `CodexAdapter`。
- 调用：`codex exec --json --dangerously-bypass-approvals-and-sandbox` 子进程，支持 `exec resume <thread_id>` 续接。
- 输出解析：逐行解析 JSON，按 Codex CLI 事件类型归一化：`thread.started→init`、`item.started(command_execution)→tool_call`、`item.completed(agent_message→text、reasoning→text+[thinking]前缀、command_execution→tool_result)`、`turn.completed→done`。归一化后的 `StreamEvent` 由后端透传至前端，前端直接按事件类型渲染，无需二次映射。

### 4.4 OpenCode Adapter

- 文件：`agentend/src/adapters/opencode.py`，类 `OpenCodeAdapter`。
- 调用：`opencode run <prompt> --format json` 子进程，支持 `--session` + `--fork` 续接、`--model` 与 `--agent` 选择。
- 输出解析：逐行解析 NDJSON，按 OpenCode CLI 事件类型归一化：`step_start→init`、`text→text`、`reasoning→text+[thinking]前缀`、`tool_use→tool_call`（error 状态产出 `tool_result`）、`error→error`。归一化后的 `StreamEvent` 由后端透传至前端，前端直接按事件类型渲染，无需二次映射。

### 4.5 Adapter Registry

- 文件：`agentend/src/adapters/registry.py`，类 `AdapterRegistry`。
- 提供 `register(agent_type, adapter_cls) / get(agent_type) / list()`。
- 在 `app/main.py` 的 lifespan 中通过 `create_adapter_registry()` 完成四类注册。
- **可插拔**：新增 Agent = 新增一个 Adapter 文件 + 在 Registry 注册一行，前端与后端零改动。

> **Orchestrator**（见 §5）也注册为一种 Adapter，但它**不拉起子进程**，而是驱动内部 LangGraph + ExecutionEngine，故不归入「编码 Agent」一类。

### 4.6 Skill 体系设计

技能是「按需下发到 Agent 工作区的能力包」，统一由 `SkillHub` 管理（`backend/internal/model/skill.go`）：

- **内置技能（builtin）**：`taskctl`（自动提交 / 合并管理）、`render`（HTML / 图片 / 附件卡片渲染）。由 Agent 端 `agentend/src/skills/provisioner.py` 在 Agent 启动时下发到工作区，存文件系统，不可删改。
- **外置技能（external）**：用户上传压缩包，存 **MySQL `longblob`**（`SkillHub.Content`），校验总大小 ≤ 10MB、文件数 ≤ 100、路径合法性（防穿越）、含 `SKILL.md` + YAML frontmatter、命名不与内置冲突。
- **导入**：物理拷贝 hub 副本到目标 Session 工作区 `{worktree}/{config_dir}/skills/<name>/`，并写 `agent_skill` 关联表。删除 hub 源不影响已导入副本。
- **适用范围**：仅编码 Agent（claude-code / opencode / codex），Orchestrator 不参与技能导入。

详见 SkillsHub 外置技能设计文档与 Skills 数据库迁移文档。

## 5. Runtime 架构设计

> Orchestrator 是唯一不拉子进程的「Adapter」：它驱动 LangGraph 规划引擎 + ExecutionEngine 波次执行，是群聊编排的核心。设计基准：**冻结 5——LangGraph 只是 Planner Engine，不接管 runtime state**。

### 5.1 Runtime 组成

| 组件 | 位置 | 职责 |
|------|------|------|
| LangGraph 状态机 | `agentend/src/orchestrator/planning/` | 规划 / 审查 / 分派 / 复盘 / 记忆 |
| ExecutionEngine | `agentend/src/orchestrator/execution/engine.py` | 波次内并行执行（`asyncio.gather`）、单任务超时管控、worktree 自动创建 |
| Dispatcher | `agentend/src/orchestrator/execution/dispatcher.py` | 将规划拆解为 `DispatchResult` 列表，经 `topological_sort` 按 `depends_on` 拓扑排序成波次 |
| RuntimeState / TaskState | `agentend/src/orchestrator/execution/state.py` | `TaskState` 枚举（单任务态 PENDING/RUNNING/COMPLETED/FAILED）；`RuntimeState` 为多任务状态容器 |
| OrchestratorAdapter | `agentend/src/adapters/orchestrator.py` | 驱动完整编排闭环：LangGraph 执行 + 波次迭代 + Replan 循环（`max_iterations=3`） |
| SessionManager | `agentend/src/session/manager.py` | 内存会话表 + 状态机校验 |
| BackendClient | `agentend/src/clients/backend_client.py` | 跨 Agent 记忆查询（`get_agent_window_messages`）、触发子 Agent（`run_task`）、SSE 订阅子 Agent 输出（`stream_result`）、置顶公告查询 |

LangGraph 节点链路：

```
skill_prepare → reason → human_review → dispatch → execute → review → evolve → save_mem
```

- `skill_prepare`：L1 技能发现 + 编排 Prompt 构建（Agent 描述 + 编排经验 + Pin 上下文）。
- `reason`：意图理解，产出规划（plan）或直接文本（text）。
- `human_review`：规划产出后**强制阻塞**等待用户审查（人工介入点，见 §14）。
- `dispatch`：将规划拆解为子任务列表并经拓扑排序成波次。
- `execute`：图中为 placeholder 节点，**实际执行由 OrchestratorAdapter 在图外驱动**——迭代波次、调 ExecutionEngine 并行执行子 Agent。
- `review`：聚合子任务结果；若检测到失败则置 `needs_replan=True`，路由回 `skill_prepare` 进入 Replan 循环。
- `evolve / save_mem`：复盘经验（EvolutionStore）、沉淀编排记忆（ConversationMemoryStore）。

### 5.2 Runtime 编排闭环

```
OrchestratorAdapter.stream_chat() 驱动完整闭环：

┌─ LangGraph 单次执行 ─────────────────────────────────────────┐
│  skill_prepare → reason → human_review ─(approve)─▶          │
│  dispatch → execute(placeholder) → review                    │
│       └─ needs_replan? ──yes──▶ 返回 skill_prepare（Replan） │
│       └─ needs_replan? ──no───▶ evolve → save_mem → END      │
└──────────────────────────────────────────────────────────────┘
        ↓ graph 终止后，Adapter 在图外执行：
        ↓   1. topological_sort → 拆波次
        ↓   2. 逐波次调 ExecutionEngine.execute()（波内并行、波间串行）
        ↓   3. 若有失败任务 → 构建 replan 消息、重新执行 graph（≤3 轮）
```

- 会话状态迁移由 `SessionManager` 经 `_VALID_TRANSITIONS` 校验（见 §7.1）。
- 终态触发 `runtime_completed` / `error` 事件。

### 5.3 Runtime 上下文管理

- **跨 Agent 记忆**（详见跨 Agent 记忆设计文档）：子 Agent 执行前，后端经 `GET /tasks/:id/messages/window` 查询「当前 Task 内其他 Session 在 T 时刻之后、状态为 completed/streaming 的消息」，每条经 `truncateWindowMessage` 截断 2000 字；Agent 端通过 `BackendClient.get_agent_window_messages` 获取后，经 `build_group_chat_context()` 格式化为 Prompt 注入 `orchestrator_context`。
- **编排记忆**：`ConversationMemoryStore`（对话历史）+ `EvolutionStore`（编排经验）落 `shared/.agent/memory/`。
- **窗口隔离**：波次串行天然保证——波次 N 的子任务执行时，前序波次消息已全部落库，可被记忆查询覆盖。

### 5.4 Runtime 隔离机制

- **进程隔离**：每个编码 Agent 任务拉起独立子进程；Session 与进程一一对应。
- **工作区隔离**：每 Session 独立 git worktree（见 §6），由 `ExecutionEngine._ensure_worktree()` 在任务执行前自动创建。
- **波次隔离**：波次间由 OrchestratorAdapter 逐波迭代保证串行——前波全部完成（含消息落库）后才启动后波；波次内通过 `asyncio.gather` 并行执行，避免并发写冲突。

## 6. Workspace 设计

### 6.1 Workspace 结构

每个 Task 在仓库旁创建一组以 `task_id` 为根的隔离目录，所有子目录（任务基线、Agent 工作区、共享区）均嵌套其下：

```
<repo>/../worktrees/{task_id}/
├── task-base/                           # 任务基 worktree（共享基线）
│                                        #   branch: task/{task_id}, base: main
├── {session_id_1}/                      # Agent 1 独立 worktree
│   └── {config_dir}/                    # Agent 配置目录（如 .claude / .opencode）
│       ├── skills/                      #   内置 + 外置技能（物理拷贝至此）
│       └── SOUL.md                      #   非 Orchestrator 的身份文档
├── {session_id_2}/                      # Agent 2 独立 worktree
│   └── ...
└── shared/.agent/                       # 任务级共享区
    ├── SOUL.md                          #   Orchestrator 身份文档
    ├── plans/                           #   规划产物
    ├── memory/                          #   编排记忆
    │   ├── common/                      #     跨 Agent 共享记忆
    │   └── {session_id}/                #     各 Agent 独立记忆
    └── config.yaml                      #   taskctl 消费配置
```

### 6.2 Git Worktree 方案

- 实现：`agentend/src/workspace/git_ops.py`，`worktree_add(repo_path, path, branch, base_branch)`。
- 分支命名：任务基线 `task/{task_id}`（基于 `main`）；Agent 分支 `agent/{session_id}/{task_id}`（基于任务基线分支）。
- 优势：多 Agent 共享同一仓库对象库（省空间）但拥有独立工作目录与分支，互不污染；支持独立 commit / revert / merge-to-main。
- 恢复：`agentend/src/workspace/recovery.py` 在启动时扫描并修复孤儿 worktree。

### 6.3 Shared 目录设计

`shared/.agent/`（完整路径 `{worktrees}/{task_id}/shared/.agent/`）是任务级共享上下文：跨 Agent 记忆、Orchestrator 规划与身份、编排经验均落此地，供同 Task 内所有 Agent 读取，实现「后执行的 Agent 看到前序 Agent 的工作」。共享目录由 `SkillProvisioner.init_shared_dirs()` 在 Agent workspace 创建时初始化，Orchestrator 通过 `_orchestrator_kwargs()` 推算路径并写入 SOUL.md 与 plans。

### 6.4 Artifact 管理

遵循 **冻结 4——Artifact 只是任务产物的引用**：不存内容、不做版本 / 血缘。产物（Diff / HTML / 图片）按 Session 路径内联引用，前端按卡片类型渲染；Diff 通过 `diff_snapshot` 表存快照供应用前二次确认。

## 7. 状态机设计

> 所有状态枚举与合法转换由契约 `session-state.yaml` 单一定义，`make generate` 同步三端，禁止手工漂移。

### 7.1 Session 状态机

```
              running ─────────────── awaiting_review
            ▲  │  │  │                      │
            │  │  │  └──────────────────────┘ (approve / discuss 都回 running)
   create ──┘  │  │
            idle│  │
              completed* │
              interrupted*
              error*
              inactive*   ← 仅由外部 API 设置（用户停用）
```

| 状态 | 含义 | 终态 | 前端渲染 |
|------|------|------|----------|
| `idle` | 空闲，等待任务 | 否 | 无特殊视觉标记（默认态） |
| `running` | 执行中 | 否 | ✅ Agent 头像旋转 + 聊天区「正在回复...」 |
| `awaiting_review` | 等待用户审查 Orchestrator 规划 | 否 | ✅ 触发 PlanReview 审查卡片（无文字标签） |
| `completed` | 正常完成 | 是 | ✅ RuntimeStatus 卡「完成」 |
| `interrupted` | 被中断 | 是 | 契约定义存在，前端无独立渲染 |
| `error` | 出错 | 是 | ✅ RuntimeStatus 卡「失败」+ TaskFailure 卡 |
| `inactive` | 用户手动停用 | 是 | Agent 从列表移除，无状态标签 |

合法转换：`idle→running`；`running→{completed, interrupted, error, awaiting_review}`；`awaiting_review→running`。

### 7.2 Task 状态机

`Task.Status`（`backend/internal/model/task.go`）：

- `active`（默认）：会话存活，参与列表与编排。
- `inactive`：用户停用（退出群聊 / 清理），会话与工作区可级联清理。

> Task 级状态更轻量——复杂执行态收敛到 Session（§7.1）与 Runtime（§7.3）。

### 7.3 Agent 状态机

Agent 不维护独立状态机；其「执行态」由所属 Session 承载。Agent 实体的运行态（如 CLI 子进程存活）由 `SessionManager` 在内存中跟踪，终态后回收进程。

### 7.4 Runtime 状态机

`state.py`（`agentend/src/orchestrator/execution/state.py`）定义两层结构：`TaskState` 枚举承载单任务执行态，`RuntimeState` 为多任务状态容器（以 `dict[task_id, TaskState]` 跟踪）。单任务态迁移如下：

```
PENDING → RUNNING → {COMPLETED | FAILED}
```

与 Session 状态机的关系：Runtime 完成时驱动 Session 迁移到 `completed`；失败时驱动 `error`（由 Python 唯一权威产生，符合冻结 2）。

## 8. Event 系统设计

> 事件流是三端的「血液」。本节基于契约 `event-types.yaml` 与 SSE 流式输出架构设计。

### 8.1 Event 分类

`EventType` 共 18 类，按语义分四组：

| 分组 | 事件 | 前端渲染 |
|------|------|----------|
| **基础 / 生命周期** | `init` · `text` · `done` · `error` · `heartbeat` | `text`→消息流；`done`→终态；`error`→错误提示；`init` / `heartbeat` 静默消费 |
| **工具 / 产物** | `tool_call` · `tool_result` · `artifact` | `tool_call` / `tool_result`→ToolCard；`artifact` 未处理（当前版本由工具事件承载） |
| **规划 / 审查** | `planning` · `plan_review` | `planning`→PlanCard（仅 dispatch 节点）；`plan_review`→PlanReviewCard（交互式审查） |
| **运行时 / 协作** | `runtime_executing` · `runtime_text` · `runtime_completed` · `coordination_start` · `coordination_message` · `coordination_done` · `ask_card_start` · `ask_card_done` | runtime 三件→RuntimeStatus 卡；`coordination_message` / `done`→CoordChannel；`ask_card_*`→AskAgentCard；`coordination_start` 静默 |

### 8.2 Event Schema

```yaml
StreamEvent:           # 三端统一数据结构（冻结 6：ABI 已冻结）
  type: EventType      # 事件类型（见 §8.1）
  content: object      # 事件内容，字段随 type 而异（additionalProperties: true）
  timestamp: number    # Unix 时间戳（秒，含小数）
  required: [type]
```

- `content` 按类型承载不同负载（如 `text` 携带增量文本、`plan_review` 携带波次与子任务、`artifact` 携带卡片元数据）。
- `sequence`（顺序号）不是 StreamEvent 的 schema 字段，而是运行时概念：Hub 内部维护 `HubEvent.Seq`（`atomic.Uint64`）用于 SSE 帧排序；Redis Stream 内生 ID 作为持久化序列号，由后端写入时分配，用于回放与去重（见 §8.4）。

### 8.3 Event 生命周期

```
Agent 端产出 StreamEvent
  → 后端 StreamWriter 消费（Hub 实时推送 + Redis Stream 持久缓冲 + MySQL 批量落库）
  → 前端 EventSource 接收 → useChatStream 路由 → messageStore(rAF 批处理) → 卡片渲染
```

- **终态事件**（`done` / `error`）绝不丢失：Redis Stream TTL 600s 保底 + MySQL 长期存储。
- **心跳**：后端每 15s 推 `heartbeat`，防止反向代理超时断开；10s 无新事件触发陈旧检测——查 StreamWriter 是否仍存活，若已停止则从 MySQL 兜底补齐剩余内容并发终态（`done`/`error`），而非简单回收连接。

### 8.4 SSE 流转过程（核心全链路）

后端 `backend/internal/stream/` 采用**双通道推送**架构：

```
Agent 端 SSE  ──▶  Writer.Run()  ──┬──▶  Hub（内存 Pub/Sub，~0ms）  ──▶ 前端实时
                                    ├──▶  Redis Stream（~1ms，回放/重连）
                                    └──▶  MySQL（~500ms 批量落库，长期）
```

| 机制 | 参数 | 作用 |
|------|------|------|
| Stream Key | `agent:<session>:<message>` | 按 session+message 隔离事件流 |
| 缓冲合并 | 2048 字节 或 500ms | 文本事件批量合并，降低 Redis 写入频率 |
| Stream 容量 | maxlen 10000 / TTL 600s | 短期回放窗口，溢出落 MySQL 兜底 |
| 心跳 | 15s | 保活 |
| 陈旧检测 | 10s | 10s 无新事件 → 查 Writer 存活性，从 MySQL 兜底补齐剩余内容并发终态（非僵尸连接回收） |
| 回放 | MySQL `last_seq` + Redis `XRead` | 后端从消息表读持久化断点 `last_seq`，用 Redis `XRead` 从断点补齐至最新，再切入 Hub 实时订阅 |

**三种消息状态的回放策略**（`stream_service.go`）：

| 消息状态 | 回放策略 |
|----------|----------|
| `streaming` | ① 回放 MySQL 已持久化内容（`splitContent` 500 rune 分块）→ ② Subscribe Hub → ③ Redis `XRead` 从 `last_seq` 补齐至最新 → ④ 进入 Hub 实时事件循环（心跳 15s + 陈旧检测 10s） |
| `completed` | `splitContent` 500 rune 分块回放全部内容 → 发 `done` |
| `failed` | `splitContent` 500 rune 分块回放内容 → 发 `error` |

前端以 `streamingReplay`（`messageId + offset`）游标跳过重复文本，仅追加新内容（见 §12.4）。

**Agent 切换**（群聊多 Agent 依次发言）：Writer 检测 `agent_type` / `source_messageID` 变化时，flush 当前缓冲到 MySQL、把当前子消息置 `completed`、新建一条 `streaming` 子消息——原始消息保持 `streaming` 直至本轮结束。

### 9.4 Redis 数据设计

| 用途 | 结构 | Key | 说明 |
|------|------|-----|------|
| SSE 事件缓冲 | **Stream** | `agent:<session>:<message>` | maxlen 10000 / TTL 600s；承载回放与重连 |
| 序列号 | Stream 内生 ID | — | 作为 sequence / Last-Event-ID，严格递增 |

> **应用层状态**：活跃 StreamWriter 实例由 Go 进程内 `sync.Map`（`stream/writer.go`）按 messageID 跟踪，用于查询 Writer 存活性与陈旧检测。此为进程内存状态，非 Redis 数据。

> Redis 在本系统**只做流式缓冲**，不承担业务状态存储——业务权威在 MySQL（符合冻结 3：不做持久化队列语义，仅借用 Stream 的有序回放能力）。

## 11. 安全设计

### 11.1 权限控制

- 单人系统，无多租户；管理后台以独立密码二次验证（`POST /admin/auth`，5 次/分钟限流）。
- 后台敏感配置查看（如 Agent YAML）需再次密码验证；`adminToken`（JWT，1 小时过期）写 store 后会话内免重复。
- 所有 Agent 能力访问**必须经后端中转与认证**，前端不直连 Agent 端。

### 11.2 Workspace 隔离

- 每 Agent 独立 git worktree + 分支，物理隔离工作目录（§6）。
- **不向客户端暴露工作区绝对路径**：所有文件读写经 `/workspace/:id/files/*` 或 `/session/:id/files/*` 间接寻址。

### 11.3 Agent 隔离

- 每 Session 独立子进程与上下文；进程在终态后回收。
- 群聊多 Agent 通过波次屏障与 worktree 分支隔离，避免并发写冲突。
- Orchestrator 工具调用权限沿用 Agent 自身能力（平台层不做额外沙箱），但在**审查入口**保留人工把关（规划审查 + Diff 应用二次确认）。

### 11.4 输入校验

| 输入 | 校验 | 层 |
|------|------|----|
| SOUL.md | ≤ 300 字 | 后端（GORM size:300） |
| 技能上传 | 总大小 ≤ 10MB、文件数 ≤ 100、路径防穿越、含 SKILL.md、命名不冲突 | 后端 |
| 技能命名 | 不与内置冲突 | 后端 |
| @ 路由 | 未知 Agent 前端即时提示 / 后端 400 拒绝，不静默派错 | 前后端 |
| Repo 路径 | 合法性与可初始化校验（契约 `validate-repo-path.yaml`） | Agent 端 |

## 12. 性能设计

### 12.1 流式输出优化（三端）

| 端 | 策略 |
|----|------|
| Agent 端 | 异步子进程 + `AsyncIterator` 流式产出，避免全量缓冲 |
| 后端 | 文本事件 2048B / 500ms 批量合并写 Redis；Hub 内存直推零延迟 |
| 前端 | **rAF 批处理**：token 入 `_textBufs`，`requestAnimationFrame` 每帧 flush 一次，杜绝逐字重渲染 |

### 12.2 Redis 优化

- Stream maxlen 10000 + TTL 600s 双保险，控制内存增长。
- 合并写入降低 QPS；仅流式缓冲，不承担业务状态，避免大 Key。
- 陈旧连接 10s 检测回收，防泄漏。

### 12.3 MySQL 优化

- 关键查询路径建索引：`message(session_id, status)`、`message(session_id)`、`message(group_id)`。
- 流式落库批量（500ms）而非逐条，降低写入压力。
- 外置技能用 `longblob` 单字段存储，避免多文件表关联。

### 12.4 前端渲染优化

| 策略 | 实现 |
|------|------|
| rAF 批处理 | `messageStore` 每帧 flush 一次文本缓冲 |
| 虚拟列表 | `MessageList` 基于 `@tanstack/react-virtual`，≥50 条消息时启用虚拟渲染，overscan=5 |
| 流式光标 | 闪烁光标提示生成中 |
| 智能滚动 | 底部自动跟随，上滚停止跟随 + 「回到底部」 |
| 断线去重 | `streamingReplay(messageId+offset)` 跳过重叠，仅追加新内容 |

**渲染管线（6 层）**：`EventSource → useChatStream(路由) → messageStore(rAF) → MessageList(虚拟) → MessageBubble/BlockRenderer → 具体 Card`。

## 14. 技术难点与解决方案

### 14.1 Multi-Agent 协作

| 难点 | 方案 |
|------|------|
| 意图理解 / 拆解 | LangGraph 规划引擎，`plan_and_dispatch` 工具产出波次 |
| 并行 / 串行编排 | ExecutionEngine 波次模型：波次内 `asyncio.gather` 并行、波次间串行屏障 |
| 结果聚合 | `review / evolve` 节点聚合 + `final_summary` 卡汇报 |
| 后执行不重复 | 跨 Agent 记忆：后置 Agent 注入前序发言（每条截断 2000 字） |

### 14.2 Git 冲突处理

- 每 Agent 独立 worktree + 分支从源头上隔离写冲突。
- 冲突检测后由 `merge-to-main` 流程给出回滚建议（PRD §11.2，P2）。

### 14.3 Runtime 调度

- 波次串行天然隔离；通过 `ask_agent` 工具支持 Agent 间反向提问（`ask_card_*` 事件）。
- 审查节点 `human_review` 以 `asyncio.Event` 阻塞，由 `POST /tasks/:id/review` 唤醒，实现「暂停等审批」。

### 14.4 流式传输

| 难点 | 方案 |
|------|------|
| 连接 = 生命周期 | Redis Stream 持久缓冲解耦；重连按 Last-Event-ID 续传 |
| 不丢不重 | sequence 严格递增 + 前端 `streamingReplay` 去重 |
| 长连接被代理掐断 | 15s 心跳 + 等待期保活 |
| 多 Agent 依次发言 | Writer 检测 agent 切换，flush+建子消息，原始消息维持 streaming 至本轮结束 |

## 15. 技术风险

### 15.1 已知限制（本期遗留）

| 模块 | 限制 | 影响 |
|------|------|------|
| 编排 | 仅「波次」串/并行，非完整 DAG | 复杂依赖图表达受限 |
| 存储 | 事件以消息表持久化，非按序号完整事件日志 | time travel / 全量回放受限 |
| 部署 | Agent 端单实例 | 无负载均衡 / 横向扩展 |
| 多端 | 仅 Web | 桌面 / 移动未做 |

### 15.2 潜在风险

| 风险 | 概率 | 影响 | 应对 |
|------|------|------|------|
| Orchestrator 误判（派错 / 拆解不当） | 中 | 严重 | 规划审查强制人工介入；@ 直达绕过编排 |
| Agent 异常退出 / 超时 | 中 | 中 | Replan 重新规划（≤3 轮） + `TaskFailureCard` 明确告知 |
| 高频工具输出堵塞流式通道 | 低 | 中 | 流量整形 + 关键事件优先（§12） |
| Go / Python 状态漂移 | 低 | 严重 | 冻结 2：仅 Python 产执行迁移，Go 请求 Python 自终结 |
| 长连接被代理超时断开 | 中 | 中 | 心跳保活 |

### 15.3 应对策略（架构纪律）

守住「务实优先、不过度设计」，重申 §2.3 冻结边界：后端不做执行引擎、EventBus 不做队列、Artifact 不做资产平台、LangGraph 不做 Runtime Kernel。任何「升级冲动」先对照冻结清单评估。
