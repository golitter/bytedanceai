# 设计审计 Prompt

对 { xxx }下的计划文档进行审计，发现问题后直接修正。

## 审计流程

### 第一阶段：审查

逐项检查以下维度：

- **结构完整性** — 是否包含背景、目标、方案、影响范围等必要章节
- **与现状一致性** — 方案中引用的代码路径、模块名、接口签名是否与代码库实际匹配
- **架构合规** — 是否遵循项目既有架构约定（三层分离、契约优先、AGENTS.md 规范等）
- **可行性** — 方案步骤是否可执行，有无遗漏前提条件或依赖
- **边界与风险** — 是否说明了异常场景、降级策略、对现有功能的影响

输出审查发现，每条包含：
- 严重程度（P0 信息缺失 / P1 与代码不符 / P2 表述模糊 / P3 建议）
- 文档位置（章节/段落）
- 问题描述
- 修正建议

### 第二阶段：修正

1. 按严重程度排序，从 P0 开始逐一修正
2. 修正时对照实际代码验证引用的路径、类型、接口名
3. 只改文档本身，不改代码
4. 汇总修正结果
