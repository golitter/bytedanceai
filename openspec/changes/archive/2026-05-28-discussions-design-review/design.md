## Context

discussions/ 目录包含 4 篇 Phase 5 核心设计文档，覆盖 Conversation Layer、Group Chat 实现、项目现状和 Orchestrator 功能清单。这些文档将在 Phase 5 实施中作为指导，但尚未经过与实际代码库的交叉验证。

审查范围：将文档中的设计决策、数据模型、事件定义、文件路径、API 变更等与前端、后端、AgentEnd 的实际代码逐一对照。

## Goals / Non-Goals

**Goals:**
- 识别文档间矛盾和与代码的不一致
- 标注设计盲点和缺失的实施方案
- 输出可操作的问题清单到 `discussions/design-review-issues.md`

**Non-Goals:**
- 不修改现有代码
- 不重新设计架构
- 不创建新的 AGENTS.md 文档

## Decisions

### 1. 审查维度

按以下维度逐一检查：
- **跨文档一致性**：4 篇文档之间的术语、命名、数据模型是否统一
- **代码一致性**：文档描述与前端/后端/AgentEnd 实际代码是否匹配
- **契约层合规**：新设计是否遵循 contracts-first 原则
- **职责边界**：三端职责划分是否与 AGENTS.md 声明一致
- **实施可行性**：设计是否有遗漏步骤或隐含依赖

### 2. 问题分级

- **P0（阻塞）**：相互矛盾的设计决策，不解决就无法开始实施
- **P1（重要）**：影响实施质量但不阻塞进度的问题
- **P2（建议）**：改进建议，可后续处理

### 3. 输出格式

在 `discussions/design-review-issues.md` 中按分级列出每个问题，包含：问题描述、涉及文档、涉及代码文件、建议修复方案。

## Risks / Trade-offs

- **审查主观性** → 每个问题附带具体的代码/文档引用，避免模糊表述
- **文档修正范围** → 本次只输出问题清单，不直接修改原始设计文档，由作者决定是否采纳
